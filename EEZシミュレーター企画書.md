# EEZシミュレーター 企画書

Claude Codeへ: この企画書に従って開発してください。フェーズ1から順に実装し、各フェーズの受け入れ基準を満たしてから次に進むこと。

---

## 1. 概要

日本の排他的経済水域(EEZ)を地図上に可視化し、島をドラッグして動かすとEEZがリアルタイムに再計算される**教育用Webアプリ**。

「沖ノ鳥島のような小さな島が国土面積を超えるEEZを生む」ことを、実際に触って体感できるのが核となる価値。

- **ターゲット**: 中高生〜一般(地理・公民の学習、教養)
- **プラットフォーム**: Webアプリ(SPA)。デスクトップ優先、タブレット対応
- **言語**: 日本語UI

## 2. コア機能(優先順)

1. **EEZの可視化**: 日本周辺の地図上に日本のEEZを塗りで表示。近隣国(韓国・中国・台湾・ロシア・フィリピン等)のEEZも別色で表示
2. **寄与の可視化**: 島や沿岸をホバー/選択すると、その基点が生み出しているEEZ範囲をハイライト。「この島がなければEEZはこれだけ縮む」を表示できる
3. **島ドラッグシミュレーション**: 主要な島(沖ノ鳥島、南鳥島、与那国島、択捉島など＋本州等の主要陸地)をドラッグで移動でき、EEZ・中間線がリアルタイムに再計算される
4. **島のON/OFF**: 島を「消す」トグル。沖ノ鳥島を消すとEEZ面積がどれだけ減るかを即座に表示
5. **面積表示**: 日本のEEZ総面積(km²)を常時表示し、操作に応じて増減をアニメーション表示。国土面積(約37.8万km²)との比較表示
6. **リセット**: ワンクリックで現実の状態に戻す

## 3. EEZ計算のルール(ドメイン知識)

- EEZは基線(海岸線)から**200海里(370.4km)**
- 領海は基線から12海里。EEZ表示と区別する(領海はフェーズ2以降でよい)
- 隣国と200海里が重複する海域は、**両国の基線から等距離の中間線**で分割する(これがシミュレーションの核)
- 国連海洋法条約121条: 「人間の居住又は独自の経済的生活を維持できない岩」はEEZを持たない。沖ノ鳥島に関する各国の立場の違いは教育コンテンツとして脚注表示する(アプリとしては日本の立場=EEZありをデフォルトとする)

### 計算アルゴリズム

1. 各国の基線を点群としてサンプリング(海岸線ポリゴンから間隔一定で抽出。精度と性能のバランスは実装時に調整、目安5〜20km間隔)
2. 各点から200海里の測地バッファを生成し、国ごとにunion → 各国の「200海里圏」
3. 重複海域は点群ベースのVoronoi分割(d3-delaunay等)で等距離中間線を近似し、各国に配分
4. 面積は球面面積計算(turf.area等)で算出
5. 島の移動/削除時は、その国の点群を更新して2〜4を再計算

### 精度に関する方針

- 教育目的なので**厳密な法的境界の再現は目指さない**。現実の日韓・日中間には合意済み漁業協定線や未画定海域があるが、本アプリは「純粋な等距離中間線モデル」で統一計算する
- ただし初期表示は実データ(Marine Regions)を使い、ユーザーが島を動かした時点でシミュレーションモード(等距離モデル)に切り替わる、という2モード構成を推奨。実装が複雑になる場合は最初から全て等距離モデルの自前計算でもよい(その場合、現実のEEZと多少ずれる旨をUIに明記)

## 4. データソース

| データ | ソース | 備考 |
|---|---|---|
| EEZ境界(実データ) | Marine Regions (VLIZ) World EEZ | 商用でなければ利用可。ライセンス表記必須 |
| 海岸線・陸地 | Natural Earth (1:10m) | パブリックドメイン。基線の点群サンプリング元 |
| ベースマップ | MapLibre GL JS + OpenStreetMap系タイル or Natural Earthベクタを自前描画 | APIキー不要の構成にする |

- データは事前処理スクリプト(Node.js)でGeoJSONに変換・簡略化(mapshaper/turf.simplify)し、静的ファイルとしてバンドルする。実行時に外部APIへ依存しない
- 対象範囲: 東経110°〜160°、北緯15°〜50°程度(日本の全EEZ+近隣国が入る範囲)

## 5. 技術スタック

- **フレームワーク**: Vite + React + TypeScript
- **地図**: MapLibre GL JS(deck.glは必要になったら追加)
- **地理計算**: Turf.js(buffer/union/area)、d3-delaunay(Voronoi)
- **再計算の実行**: Web Worker(ドラッグ中のUIブロッキング防止)。ドラッグ中は低解像度点群で高速プレビュー、ドロップ時に高解像度で確定計算
- **状態管理**: 軽量でよい(Zustand等)
- **テスト**: Vitest。地理計算ロジック(バッファ、Voronoi配分、面積)は必ずユニットテストを書く
- **デプロイ**: 静的ホスティング前提(GitHub Pages / Cloudflare Pages)

## 6. UI構成

```
+----------------------------------------------------------+
| ヘッダー: タイトル / リセットボタン / モード表示            |
+------------------------------------------+---------------+
|                                          | サイドパネル    |
|              地図(メイン)                 | ・日本EEZ面積   |
|   - EEZ塗り分け(国別配色)                 |   (増減表示)    |
|   - ドラッグ可能な島マーカー               | ・島リスト      |
|   - 中間線の表示                          |   (ON/OFFトグル)|
|                                          | ・選択中の島の  |
|                                          |   寄与EEZ面積   |
+------------------------------------------+---------------+
| フッター: データ出典 / 免責(境界は簡略モデルである旨)        |
+----------------------------------------------------------+
```

- 配色: 日本=青系、各国は識別しやすい別色。係争・重複海域は斜線または中間色
- 島マーカーはドラッグ可能であることが分かるUI(ハンドル表示)
- 面積はカウントアップ/ダウンのアニメーションで変化を強調

## 7. 教育コンテンツ

- 各島の情報カード: 名前、実面積、生み出すEEZ面積、豆知識(例: 沖ノ鳥島は満潮時に数m²だが約40万km²のEEZを支える)
- 「もしも」プリセットシナリオ(フェーズ3): 「沖ノ鳥島がなかったら」「南鳥島がなかったら」等をワンタップで再現

## 8. センシティブな領域の扱い(必須要件)

- 竹島・尖閣諸島・北方領土の周辺海域は「係争中/未画定」の専用スタイル(斜線等)で表示し、特定の立場を断定しない
- 免責文をフッターに常設: 「本アプリは教育目的の簡略モデルであり、法的な境界を示すものではありません」
- デフォルト表示は日本の公式見解ベースとしつつ、上記の注記を必ず添える

## 9. 開発フェーズと受け入れ基準

### フェーズ1: 静的可視化(MVP最小)
- 地図上に日本と近隣国のEEZ(実データ)が色分け表示される
- 日本のEEZ総面積が表示される
- **受け入れ基準**: ブラウザで地図をパン/ズームでき、EEZ表示に体感的なもたつきがない

### フェーズ2: 計算エンジン
- 基線点群→200海里バッファ→Voronoi中間線→面積算出のパイプラインをWeb Workerで実装
- 自前計算による日本EEZ面積が公称値(約447万km²、領海含む区分に注意)と**±10%以内**で一致
- **受け入れ基準**: 上記精度＋計算ロジックのユニットテストが通る

### フェーズ3: インタラクション
- 島のドラッグ移動、ON/OFF、リセット、面積の増減アニメーション
- 沖ノ鳥島OFFで面積が約40万km²減ることが確認できる
- **受け入れ基準**: ドラッグ中のプレビュー更新が概ね30fps以上/確定計算2秒以内

### フェーズ4: 教育コンテンツ+仕上げ
- 島情報カード、プリセットシナリオ、係争海域スタイル、免責表示、モバイル対応
- **受け入れ基準**: 初見の中学生が説明なしで「島を動かすとEEZが変わる」ことに気づけるUI

## 10. スコープ外(作らないもの)

- ユーザーアカウント、データ保存、共有機能
- 大陸棚延長(200海里超)の扱い
- 漁業協定線など個別条約の再現
- ネイティブアプリ版

## 11. リスクと対策

| リスク | 対策 |
|---|---|
| Voronoi中間線の精度不足 | 点群密度を上げる/境界付近のみ再サンプリング。教育目的なので±10%許容 |
| ドラッグ中の再計算が重い | 低解像度プレビュー+Worker。それでも重ければ影響範囲(移動した島の周辺)のみ差分再計算 |
| turf.bufferの高緯度歪み | 測地バッファ実装(geodesic-buffer系ライブラリ or 自前実装)に差し替え可能な設計にする |
| Marine Regionsデータが重い | mapshaperで簡略化し、ズームレベル別に出し分け |
